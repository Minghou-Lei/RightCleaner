# MIN-39 MVP 功能边界与非目标

## 背景

`MIN-39` 的目标是为 RightCleaner 首版定义足够清晰的产品边界，保证团队先交付一个可验证、可使用、可迭代的最小产品，而不是在“清理一切”的方向上持续扩张范围。

首版聚焦一个核心闭环：

1. 发现可清理对象
2. 呈现清理建议与风险信息
3. 由用户确认后执行清理
4. 返回可理解的结果反馈

只要不能直接支撑这个闭环的能力，都不应该进入 MVP。

## MVP 成功标准

首版上线时，产品必须满足以下标准：

- 用户可以在一次会话中完成“扫描 -> 选择 -> 确认 -> 清理 -> 查看结果”的完整流程。
- 用户能够理解每个清理项的大致来源、占用空间或价值、以及执行清理的后果。
- 清理动作默认是保守且可控的，不允许在未确认的情况下执行高风险删除。
- 首版只解决高频、低歧义、容易判断的清理场景，不承担“全盘系统优化”承诺。

## 首版必须支持的能力

### 1. 基础扫描能力

首版必须支持对明确范围内的常见冗余项进行扫描，并输出结构化结果。

必须支持：

- 扫描预定义的清理目录或类别，而不是要求用户手工逐路径配置。
- 识别高频且低风险的可清理对象，例如缓存、临时文件、回收站类内容、日志残留或安装过程遗留文件。
- 为每个候选项提供至少以下信息：
  - 名称或路径
  - 分类
  - 预计可释放空间或大小
  - 基础风险级别

MVP 不要求做到“识别所有垃圾文件”，但必须让用户在首轮体验中看到真实、可操作的结果。

### 2. 清理建议与决策支持

首版必须让用户知道“为什么建议清理”和“是否值得清理”。

必须支持：

- 按类别或风险进行分组展示。
- 展示每类项目的数量与可释放空间汇总。
- 对高风险或不确定项给出明确提示，避免与普通缓存项混在一起。
- 支持用户选择全部、按类别选择、按单项选择。

首版的重点是让决策过程足够清晰，而不是做复杂的推荐算法。

### 3. 安全可控的执行流程

首版必须将“安全”放在“自动化”之前。

必须支持：

- 执行前二次确认，明确展示本次将被清理的范围。
- 对执行失败的项目进行逐项反馈，而不是整体静默失败。
- 在清理结束后展示成功、失败、跳过的结果汇总。
- 对明显高风险的系统关键位置默认不纳入可清理集合。

如果某类内容是否安全存在明显争议，则不应进入首版默认清理能力。

### 4. 最小可用的结果反馈

首版必须让用户理解收益，形成闭环。

必须支持：

- 展示本次已释放空间。
- 展示清理耗时或处理完成状态。
- 展示失败项，便于用户后续重试或手动处理。

首版不需要复杂的数据分析仪表盘，但必须有最基本的结果反馈页面或结果区。

## 明确不做的功能

以下能力属于非目标，不进入首版：

- 智能诊断整个系统性能问题。
- 自动修复注册表、驱动、系统服务或其他高风险系统配置。
- 无人值守的一键深度清理。
- 云端账号、跨设备同步或远程管理。
- 多角色协作、团队版管理后台。
- 高级调度能力，例如复杂定时任务、策略编排、条件触发。
- 个性化推荐模型、机器学习排序、基于历史行为的智能预测。
- 全量文件去重、照片整理、下载库治理等独立产品级能力。
- 面向企业合规或审计场景的报表系统。
- 国际化、多品牌皮肤、插件生态等扩展型能力。

这些方向不是没有价值，而是与首版验证目标不一致，过早纳入只会拖慢交付。

## 版本边界原则

如果一个需求同时满足以下任意一项，应默认判定为不属于 MVP：

- 不能直接提升“扫描 -> 选择 -> 确认 -> 清理 -> 反馈”主流程的完成率。
- 需要新增复杂系统权限、高风险底层改动或难以解释的自动化行为。
- 需要长期数据积累才能体现价值。
- 主要服务于低频、边缘或专家型用户。
- 会显著增加测试矩阵，但对首版核心体验没有直接增益。

## 迭代优先级

### P0：首版必须交付

- 预定义低风险清理范围
- 扫描结果列表与分类汇总
- 单项或分类选择能力
- 执行前确认
- 安全清理执行
- 清理结果反馈

### P1：MVP 验证后优先扩展

- 更多可清理类别，但必须保持低歧义
- 更细粒度的风险提示与解释文案
- 清理历史记录
- 回收站式兜底或有限恢复能力
- 基础定时提醒，但不自动执行

### P2：后续视验证结果再决定

- 自动化清理策略
- 更深层的系统优化能力
- 个性化推荐与智能排序
- 账户体系、同步、云能力
- 高级运营与分析面板

## 需求准入标准

后续新增需求进入版本排期前，必须回答以下问题：

1. 它是否直接增强首版核心闭环？
2. 它是否会引入新的高风险删除面？
3. 它是否要求额外的复杂配置、学习成本或系统权限？
4. 如果本期不做，是否会阻断用户完成核心任务？

如果第 4 个问题答案是否定的，这项需求通常不应挤进当前版本。

## 决策结论

RightCleaner 首版不是“全能系统清理平台”，而是“帮助用户安全清理高频低风险冗余内容”的最小可用产品。

产品判断标准应始终围绕以下一句话：

> 先把高频、低风险、可解释、可确认的清理流程做扎实，再考虑更深、更广、更自动化的能力。
