# MIN-42 技术选型：桌面框架、语言与权限模型

## 1. 目标

为 RightCleaner 首版明确以下技术决策：

- 桌面应用框架选型
- UI 技术栈
- 底层系统操作实现方式
- 权限边界与风险控制模型

本结论服务于 `MIN-39` 的 MVP 边界：先交付一个安全、清晰、可解释的 Windows 桌面清理工具，而不是一开始就做成高权限、重后台、全能系统优化平台。

## 2. 候选方案对比

| 方案 | 优点 | 问题 | 结论 |
|---|---|---|---|
| WinUI 3 + C#/.NET | 原生 Windows 体验最好；与 Windows App SDK 一致；系统集成自然 | UI 研发速度相对慢；组件生态和设计迭代灵活性不如 Web UI；前后端职责更容易混在一起 | 备选，不作为首版主方案 |
| Electron + TypeScript | Web 技术门槛低；生态成熟；桌面经验广泛 | 运行时体积和内存占用偏大；Chromium + Node 攻击面更大；需要更严格的 preload / IPC 安全治理 | 不选 |
| Tauri 2 + React + TypeScript + Rust | 包体和内存更克制；前端与系统能力边界清晰；Rust 适合做受限命令层；权限模型可细化到 capability / command | Windows 原生控件一致性不如 WinUI 3；团队需要同时维护前端和 Rust | 选用 |

## 3. 最终结论

RightCleaner 首版采用：

- 桌面框架：`Tauri 2`
- UI：`React + TypeScript`
- 底层系统能力：`Rust`
- Windows 平台访问：优先通过 `windows` crate 调用 Win32 / Windows Shell API，必要时辅以成熟 Rust crate

这套组合最符合当前产品阶段的三个要求：

1. UI 迭代要快，便于把 `MIN-40` 的信息架构快速落成桌面界面。
2. 底层操作要可控，不能把文件系统、注册表、Shell 行为直接暴露给前端。
3. 首版要保持低权限、低常驻复杂度，避免一开始引入服务进程、全局提权和不可解释行为。

## 4. 为什么不选 WinUI 3

WinUI 3 的优势是非常明确的：它是 Windows App SDK 的现代原生 UI 方案，天然适合做 Windows 桌面产品。

但 RightCleaner 首版不把“最原生”放在第一优先级，而是把以下两点放在前面：

- 页面和交互需要高频迭代，尤其是列表、详情、批量确认、备份恢复这些复杂信息密度页面。
- 系统能力需要与 UI 严格隔离，避免 UI 层直接承担高风险操作。

如果首版选择 WinUI 3，团队会更容易把 UI、状态管理和系统操作写进同一层 C# 应用逻辑里，后续再拆边界的成本更高。相比之下，Tauri 天然要求通过命令边界访问原生能力，更适合 RightCleaner 的风险模型。

## 5. 为什么不选 Electron

Electron 的优势是成熟和通用，但不符合 RightCleaner 首版的约束：

- 对一个聚焦 Windows 清理工具的首版来说，Electron 的运行时体积和资源占用不够克制。
- Electron 需要持续严控 `preload`、IPC、上下文隔离、进程沙箱和 Node 能力暴露，治理成本不低。
- RightCleaner 不需要 Chromium + Node 的完整桌面运行时能力，也不需要依赖 Node 原生模块承接系统逻辑。

结论不是 Electron 做不到，而是它在 RightCleaner 这个问题上显得过重。

## 6. UI 技术实现方式

UI 层采用 `React + TypeScript`，承担以下职责：

- 首页、列表、详情、批量确认、备份恢复的渲染与交互
- 筛选、排序、分组、选择态、确认态等前端状态管理
- 风险提示、结果反馈、异常提示
- 调用 Tauri command 获取扫描结果、提交清理任务、读取备份信息

UI 层明确不承担：

- 直接文件删除
- 直接目录遍历
- 直接注册表读写
- 直接调用 Shell 或 PowerShell 执行系统命令

也就是说，前端只负责“表达”和“编排”，不直接持有系统权限。

## 7. 底层系统操作实现方式

底层能力放在 `src-tauri` 的 Rust 侧，按照“扫描、评估、执行、恢复”拆分模块。

建议的模块边界如下：

```text
src-tauri/
├─ src/
│  ├─ commands/
│  │  ├─ scan.rs
│  │  ├─ cleanup.rs
│  │  ├─ backup.rs
│  │  └─ settings.rs
│  ├─ domain/
│  │  ├─ candidate.rs
│  │  ├─ risk.rs
│  │  ├─ task.rs
│  │  └─ backup.rs
│  ├─ infrastructure/
│  │  ├─ fs/
│  │  ├─ shell/
│  │  ├─ registry/
│  │  └─ windows/
│  └─ main.rs
```

具体实现策略：

### 7.1 扫描

- 使用 Rust 遍历预定义目录和规则集。
- 通过 Rust 计算大小、更新时间、命中数量、候选分类。
- 扫描规则配置化，但首版只开放受控的内置规则，不开放任意脚本。

### 7.2 清理执行

- 低风险文件和目录删除由 Rust 执行。
- 默认优先走“可恢复”策略，例如进入备份链路或回收站能力，而不是直接永久删除。
- 每个候选项独立返回结果，保证失败可定位。

### 7.3 备份与恢复

- 备份元数据、原路径、任务批次信息由 Rust 统一维护。
- 恢复能力与清理能力使用相同的任务模型，避免两套状态机。

### 7.4 Windows 专有操作

Windows 专有能力优先通过 Rust 调用原生 API 完成，而不是把 PowerShell 当成主执行引擎。

优先顺序：

1. `windows` crate 封装 Win32 / Windows Shell API
2. 经过验证的 Rust crate
3. 极少数无法避免的场景，再封装单一职责的系统命令调用

这样做的原因是：

- 类型安全更强
- 错误处理更稳定
- 更容易做参数约束和单元测试
- 不把系统命令拼接暴露成默认路径

## 8. 权限模型

RightCleaner 首版采用“默认非管理员、按操作升级、前端零直权”的权限模型。

### 8.1 默认运行权限

- 应用默认以普通用户权限运行
- 不要求安装后常驻管理员权限
- 不引入长期运行的高权限后台服务

这与 `MIN-39` 一致：首版只做高频、低歧义、低风险清理场景。

### 8.2 前后端权限边界

- React 前端没有直接文件系统、注册表、Shell 权限
- 前端只能调用显式注册的 Tauri commands
- Rust commands 再调用内部受控模块

边界链路如下：

```text
UI Action
-> typed frontend API
-> Tauri command
-> Rust domain service
-> Windows / FS operation
```

### 8.3 Tauri 权限控制

Tauri 侧采用 capability-based 暴露模型：

- 仅对主窗口开放 RightCleaner 需要的 commands
- 不启用与产品无关的插件权限
- 文件系统能力按目录范围和操作类型收口
- 所有危险操作必须走显式 command，而不是开放通配式能力

### 8.4 风险分级与提权策略

把系统操作分成三层：

| 等级 | 示例 | 权限策略 |
|---|---|---|
| L1 低风险 | 用户缓存、临时目录、应用日志残留 | 普通权限可执行 |
| L2 中风险 | 需要备份确认的用户目录内容 | 普通权限执行，但必须二次确认 |
| L3 高风险 | 受保护系统目录、系统级配置、可能影响系统稳定性的操作 | 首版默认不做；后续如确有必要，再按单次提权设计 |

首版明确不做“启动即管理员”。

如果后续版本必须引入特定高权限能力，原则是：

- 只对单次任务提权，不提升整个应用生命周期
- 提权入口必须绑定明确用户动作
- 高权限逻辑与普通逻辑分离，避免所有命令都跑在管理员上下文

## 9. 与产品边界的对应关系

这个选型不是单纯偏好问题，而是直接服务产品边界：

- `MIN-39` 要求低风险、可解释、可确认，Tauri + Rust 更容易建立“前端无直权”的结构。
- `MIN-40` 有高信息密度页面，React 更适合快速构建和迭代复杂列表、筛选、分组、确认流。
- 首版不做深度系统修复和高风险自动化，因此不需要先把整套产品建立在常驻高权限模型上。

## 10. 交付约束

基于本结论，后续实现阶段应遵守以下约束：

1. 不允许在前端直接读写文件系统或注册表。
2. 不允许把 PowerShell 脚本拼接调用作为主要系统执行路径。
3. 不允许为了少数高权限场景把整个应用改成默认管理员运行。
4. 不允许在 MVP 阶段引入常驻 Windows Service。
5. 扫描与清理规则必须内置、可审计、可解释。

## 11. 决策结论

RightCleaner 首版技术选型定为：

- `Tauri 2` 作为桌面壳与命令桥接层
- `React + TypeScript` 作为 UI 实现技术
- `Rust` 作为扫描、清理、备份、恢复与 Windows 能力访问层
- `默认非管理员 + 显式命令边界 + 按需单次提权` 作为权限模型

这套方案在 Windows 产品体验、研发速度、资源占用和安全边界之间取得了当前阶段最合理的平衡。
